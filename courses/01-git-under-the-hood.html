Part 1 · the core model

Snapshots and pointers, not diffs 5 min live

The single misconception that makes git confusing: people think it stores changes. It does not. Every commit is a full snapshot of your project, and git is ruthlessly efficient about not duplicating unchanged files. A branch is not a copy - it is a movable pointer to one snapshot. HEAD is the pointer that says "you are here".

LiveCommit = snapshot, branch = pointer, HEAD = you3 min▶
  • Commit - a complete snapshot of every tracked file at one moment, with a unique hash (a3f9...), an author, a time, a message, and a link to its parent. The chain of parents IS your history.
  • Branch - a named pointer to one commit. main and feature above are just labels on snapshots. Creating a branch writes one 40-character file. That is why engineers branch constantly.
  • HEAD - the pointer to the branch (or commit) you are currently on. "Checking out" a branch moves HEAD. Almost every "wait, where am I?" moment is a HEAD question.
Real world

Linus Torvalds wrote git in 2005 in about ten days to manage the Linux kernel after the project lost access to its proprietary tool. The snapshot-and-pointer design is exactly why it scales to a project with 1,000+ contributors and a million commits: reading history is following a chain, and branching is free.

Self-studyThe three zones: working tree, staging, repository3 min read▶

Every file lives in one of three zones, and every basic command just moves files between them. This is the model that makes add and commit stop feeling arbitrary.

ZoneWhat it isCommand to enter it
Working treeyour actual files on disk, edited freely(you just edit)
Staging area (index)the guest list for the next commitgit add
Repositorypermanent snapshots, the historygit commit

Staging feels like a pointless extra step for one week, then you realize it lets you commit part of your work with a clean message while leaving the rest in progress. It is a feature, not a toll.

Part 2 · get a working git

Install, configure, or open a Codespace 5 min live

Two paths to a terminal with git. Local install is the long-term home; a Codespace is a full browser-based dev environment (git pre-installed) that gets anyone unblocked in 60 seconds - use it if install is fighting you.

LiveThe one-time setup every builder does once3 min▶

Check whether git is already there (macOS and most Linux ship it):

git --version git version 2.44.0

If missing: macOS brew install git, Windows install Git for Windows (gives you Git Bash), Linux sudo apt install git. Then set your identity once - it stamps every commit you will ever make:

git config --global user.name "Phoebe Fu" git config --global user.email "you@example.com" git config --global init.defaultBranch main
The zero-install escape hatch On any GitHub repo press . (period) or click Code → Codespaces → Create. You get VS Code in the browser with git and gh already configured and authenticated. Nothing to install. This is also how you follow along on a locked-down work laptop.
Self-studyThe gh CLI: GitHub in your terminal2 min read▶

git talks to repositories; gh (GitHub CLI) talks to GitHub - PRs, issues, releases - without leaving the terminal. You will lean on it from Session 3. Install and log in once:

gh auth login ? Where do you use GitHub? GitHub.com ? Authenticate Git with your GitHub credentials? Yes ✓ Logged in as phoebefu6

Codespaces come with gh pre-authenticated, so you can skip this there.

Part 3 · make history by hand

init, status, add, commit, log 5 min live

Five commands take a folder from "just files" to "a repository with history". Watch which zone each one touches - that is the whole point of doing it slowly once.

# turn a folder into a repo (creates the hidden .git) git init # see what git thinks - everything is untracked right now git status # stage README.md: move it to the guest list for the next snapshot git add README.md # commit: freeze the snapshot with a message in the imperative mood git commit -m "Add project README" # the history - your snapshot chain, newest first git log --oneline a3f901c (HEAD -> main) Add project README
LiveReading status and log like a local3 min▶
  • git status is your compass. Run it constantly. It tells you your branch, what is staged (green), what is modified but unstaged (red), and what is untracked. When lost, status first.
  • git add is staging, not saving. It copies the current state of a file to the staging area. Edit after adding, and you must add again - the stage holds a snapshot, not a live link.
  • git commit -m writes the snapshot. Message style: imperative mood, under ~50 chars for the summary ("Add expense parser", not "added stuff"). Future-you greps these.
  • git diff vs git diff --staged. The first shows unstaged changes (working vs stage); the second shows staged changes (stage vs last commit). Two questions, two commands.
Real world

Good commit history is a debugging superpower. When production breaks, git log and git bisect let a team binary-search 500 commits to find the exact snapshot that introduced a bug in minutes. That is only possible because each commit is a coherent, message-labeled snapshot - which starts with the habits you build today.

Self-studyConnecting to GitHub: remote, push2 min read▶

Your commits are local until you push them to a remote (GitHub). The fastest path uses gh to create the repo and wire it up in one step:

gh repo create my-project --public --source . --push ✓ Created repository phoebefu6/my-project ✓ Pushed commits to origin/main

Under the hood that sets origin (the remote's nickname) and runs git push -u origin main. Session 2 makes push/pull a daily reflex.

Build-along 2 of 3

Make your first three commits ★ 9 min · everyone types

git init, then create a README: echo "# My Project" > README.md. Run git status and read it aloud - README is untracked (red).

git add README.md, git status again - now it is staged (green). Notice the file moved zones without changing on disk.

git commit -m "Add project README". Then add two more small files (a .gitignore, a notes.md) and commit each with its own clear message. Three snapshots.

git log --oneline. You just built a history chain by hand. Read the hashes - each is a snapshot you can return to.

Push it (optional, 1 min) gh repo create my-project --public --source . --push and your local history is on GitHub. We make this routine in Session 2.

Check yourself

Three questions before you go 🎯 ◐ 90 seconds

1 · What does a commit actually store?

Git is a snapshot store, not a diff store. It shows you diffs when asked, but each commit is a complete picture - which is why branching and history reading are so fast.

2 · You edited a file, ran git add, then edited it again. What is in the next commit if you commit now?

Staging captures the file's state at add-time. The second edit sits in the working tree until you add again. This is the three-zone model doing exactly what it says.

3 · What is a branch, mechanically?

A branch is a 40-character file naming one commit. That is why engineers make them freely - Session 3 does exactly that.
